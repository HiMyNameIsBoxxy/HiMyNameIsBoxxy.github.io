PCA & iForest Analysis¶

By: Huan Shuo Hsu¶

Objectives¶

We will be employing PCA and iForest techniques for the following:

  • Detect any transactions that may be significantly different from the majority of normal transactions.
  • Detect any transactions that show unusual spending behavior compared to peer transactions within the same dataset.
  • Detect any transactions that stand out from the average (mean/median) transaction profile.
  • Detect any transactions that are anomalous when evaluated against clusters of similar transaction groups (e.g., peer groups formed by PCA components).

We will not only pinpoint suspicious transactions but also evaluate which method provides more meaningful business insights in detecting potential fraud.

Feature Engineering ¶

We will create new numeric features to capture patterns:

  • Day of week and hour of transaction (timing patterns)

  • Delay between transaction and posting (potential fraud signal)

  • Frequency encoding for Merchant Category Code (MCC) and Vendor (rare = more suspicious)

Handle missing values and scale features:

  • We will replace missing values with median

  • To ensures PCA is not dominated by one large-scale variable (Amount), we will standardize the values

PCA Method ¶

Principal Component Analysis (PCA) is a technique that reduces data into a smaller set of principal components, which capture the main directions of variance. In anomaly detection, transactions that align well with these dominant patterns are considered normal, while those that fall far outside the component space are flagged as anomalies. By setting a contamination level, PCA automatically establishes a threshold to identify the most unusual transactions.

Model Building ¶

  • n_components=5

    • Total number of principal components (PCs) to compute.
    • Captures variance across all features.

  • n_selected_components=3

    • Top 3 principal components are used for anomaly scoring.
    • Outliers are identified by their distance in this reduced subspace.

  • contamination=0.05

    • Assumes ~5% of the data are anomalies.
    • Sets the cutoff threshold for labeling outliers.

  • standardization=False

    • Data was already standardized before fitting.
    • Avoids re-standardizing inside the PCA model.

  • pca.threshold_

    • The anomaly score cutoff based on contamination.
    • Transactions with reconstruction error above this threshold are flagged as anomalies.
In [27]:
pca = PCA(
    n_components=5,             # total PCs to compute
    n_selected_components=3,    # top PCs used for anomaly scoring
    contamination=0.05,         # assume 5% anomalies
    standardization=False       # already stdized in previous code
)

Scoring & Threshold ¶

  • pca.decision_function(X_scaled)

    • Calculates anomaly scores for each transaction.
    • Higher score = more likely to be an outlier.
    • Based on the reconstruction error from the top selected principal components.

  • pca.predict(X_scaled)

    • Converts scores into binary labels:
      • 1 = anomaly (outlier)
      • 0 = normal (inlier)
    • Uses the contamination parameter (5%) to determine how many points to flag.

  • pca.threshold_

    • The cutoff score for distinguishing anomalies from normals.
    • Derived automatically from the contamination setting.
    • Any transaction with a score above this threshold is labeled as an anomaly.

Evaluation of Results ¶

  • PCA effectively captured both ends of the spectrum:
    • Extremely large transactions (e.g., multi-million dollar adjustments, high-value corporate purchases).
    • Unusually small negative transactions (e.g., refunds and credits).

  • However, PCA can sometimes overemphasize variance-heavy vendors, leading to clusters of similar anomalies (e.g., multiple large corporate purchases).

Conclusion: PCA is better at finding a wide variety of anomalies, including both massive payments and rare small reversals, making it highly flexible but potentially noisier in results.

iForest Method ¶

Isolation Forest detects anomalies by randomly partitioning the feature space with many binary trees. Points that are easy to isolate (short average path length across trees) are likely outliers; typical points require more splits. We keep the same contamination (5%) as PCA so the two methods are directly comparable in counts and thresholds.

Model Building ¶

  • decision_function(X_scaled)

    • Calculates raw anomaly scores for each transaction.
    • Higher score = more likely anomaly.
    • Based on the average path length in the isolation trees.

  • predict(X_scaled)

    • Converts scores into binary labels:
      • 1 = anomaly (outlier)
      • 0 = normal (inlier)
    • Uses contamination=0.05 so ~5% of transactions are flagged.

iforest = IForest(
    n_estimators=200,       # more trees = more stable scores
    max_samples='auto',     # min(256, n_samples) per tree

Scoring & Threshold ¶

  • iforest.threshold_
    • Cutoff score used to split anomalies vs normals.
    • For iForest in PyOD this is usually 0.0, since scores are normalized around zero.
    • Interpretation:
      • Score < 0 → normal
      • Score > 0 → anomaly

Conclusion: Isolation Forest is better at finding large-scale, high-impact anomalies that are most likely to represent financial risk, but it may miss subtle anomalies like small credits or rare reversals.
